2

Startup Phase

When you call await agent.run(), the agent begins its main loop.

The Startup Flow

await agent.run()
↓
Set _running = True
↓
await _connect_mcp()
↓
Enter main event loop
↓
while self._running:

Key Detail: Lazy MCP Connection

The _connect_mcp() method connects to MCP servers, but only if configured. It uses a one-time connection pattern:

async def _connect_mcp(self) -> None: if self._mcp_connected or self._mcp_connecting: return # ← Skip if already connected # ... connect to servers ... self._mcp_connected = True

This means MCP connections are lazy—they only happen when needed, reducing startup time.

3

Message Processing Phase (The Main Loop)

Once running, the agent continuously listens for messages and processes them concurrently.

The Main Event Loop

while self._running: try: # Wait for message with 1-second timeout msg = await asyncio.wait_for(self.bus.consume_inbound(), timeout=1.0) except asyncio.TimeoutError: continue # No message, try again if msg.content.strip().lower() == "/stop": await self._handle_stop(msg) else: # Create concurrent task for this message task = asyncio.create_task(self._dispatch(msg))

Task 1 releases lock

T=2501ms

Task 2 acquires lock (was waiting)

Task 2: Calls LLM (await provider.chat()) → pauses

T=5200ms

LLM responds to Task 2 → Task 2 resumes

Task 2: Execute web_search tool → completes

Both tasks done!

🔑 Key Insight: The Processing Lock

Notice that only one task processes at a time due to _processing_lock. This is intentional to prevent race conditions when accessing shared state (sessions, memory, tools).

However, during await points (LLM calls, tool execution), the lock is held but execution pauses, so other tasks can't steal it—they wait politely.

5

The Agent Iteration Loop (The Core Loop)

This is the heart of the agent—it loops up to max_iterations times (default: 40).

Simplified Algorithm

for iteration in range(max_iterations): # 1. Call the LLM response = await provider.chat(messages, tools, ...) # 2. Does it want to call tools? if response.has_tool_calls: # Add assistant message with tool calls messages = add_assistant_message(messages, response) # Execute each tool for tool_call in response.tool_calls: result = await tools.execute(tool_call.name, tool_call.arguments) messages = add_tool_result(messages, result) # Loop again (up to max_iterations) else: # Final response, no more tool calls final_content = response.content break

🎯 Key Takeaways

1. Concurrency Without Threads

Multiple messages are processed concurrently using async/await. When one task hits an await point (like waiting for LLM), the event loop switches to another task. This gives the illusion of parallelism on a single thread.

2. Processing Lock Prevents Race Conditions

The _processing_lock ensures only one task accesses shared state (sessions, memory) at a time. This prevents data corruption but still allows concurrency during I/O waits.

3. Task-Per-Message Design

Each incoming message gets its own async task via create_task(). These tasks are tracked in _active_tasks so they can be cancelled with /stop command.

4. Stateful Sessions

Conversation history is persisted per session. When a new message arrives, the agent loads the session history and builds context before calling the LLM.

5. Iteration Loop for Tool Use

The core iteration loop calls the LLM, checks if it wants tools, executes them, and loops up to 40 times. This enables complex multi-step reasoning.

6. Lazy Connections

MCP servers are connected lazily (on first use) rather than at startup. This speeds up initialization.

🔍 Debugging Tips

Check Active Tasks

print(agent._active_tasks) # Output: {'slack:C1234567': [, ]}

Check Session Memory

session = agent.sessions.get("slack:C1234567") print(f"Messages in session: {len(session.messages)}") for msg in session.messages[-5:]: # Last 5 messages print(f"{msg['role']}: {msg['content'][:100]}...")

Monitor Tool Execution

The agent logs all tool calls with logger.info("Tool call: {}(...)", ...). Check logs to see what tools are being used.

Check if Task was Cancelled

try: await task except asyncio.CancelledError: print("Task was cancelled (user hit /stop)")
