Nanobot Architecture

multi-channel AI agent framework — Python + asyncio + LiteLLM

1 — System Overview

Nanobot is a multi-channel AI agent framework. It connects to chat platforms (Telegram, Discord, WhatsApp, Slack, etc.), routes messages through an asynchronous bus, processes them with an LLM-powered agent loop that has access to tools and skills, and sends responses back through the originating channel.

2 — Core Architecture Layers

The system is organized into distinct layers. Chat platforms connect through Channels, which publish to a Message Bus. The Agent Loop consumes messages, builds context, calls an LLM Provider, and executes Tools. Results flow back through the bus to the channel.

4 — Agent Loop Detail

The agent loop is the brain of nanobot. It builds context from system prompts, session history, skills, and memory, then iterates: call the LLM, execute any requested tools, feed results back, and repeat until the LLM returns a final text response with no tool calls. 

        flowchart TD
          START["Inbound Message"] --> SESS["Load / Create Session"]
          SESS --> CTX["Build Context"]
          CTX --> SYS["System Prompt"]
          CTX --> HIST["Conversation History"]
          CTX --> SKILLS["Active Skills"]
          CTX --> MEMORY["Memory Files"]
          CTX --> LLM["Call LLM Provider"]
          LLM --> CHECK{"Tool calls?"}
          CHECK -->|"Yes"| EXEC["Execute Tools"]
          EXEC --> APPEND["Append results to context"]
          APPEND --> LLM
          CHECK -->|"No"| RESP["Final Response"]
          RESP --> SAVE["Save to Session"]
          SAVE --> OUT["Publish Outbound"]
Context Builder
Assembles the full message list sent to the LLM: system prompt (from AGENTS.md, SOUL.md, USER.md), active skills injected as system instructions, consolidated memory from MEMORY.md, and conversation history from the session manager. Located at agent/context.py.
Session Manager
Maintains per-conversation state keyed by channel:chat_id. Stores message history, tracks consolidation points, and provides get_history() for the context builder. Handles memory window limits. Located at session/manager.py.
5 — Channel Architecture

All channels extend BaseChannel, implementing start(), stop(), send(), and _handle_message(). The ChannelManager registers, starts, and routes outbound messages to the correct channel instance.

Channel Implementations
Telegram
python-telegram-bot, voice transcription via Whisper (Groq)
Discord
discord.py, slash commands + message handler
WhatsApp
Node.js bridge via Baileys in bridge/ dir
Slack
slack-sdk, event-driven with socket mode
Feishu / Lark
lark-oapi, webhook + event subscription
CLI
Rich + prompt_toolkit, direct agent mode
Matrix
matrix-nio, end-to-end encryption support
Email
IMAP/SMTP, poll-based inbound
DingTalk / QQ
Platform SDKs, stream-based connections
Base class pattern. Every channel inherits from BaseChannel which provides message sanitization, reconnection logic, and a consistent interface. The ChannelManager at channels/manager.py handles lifecycle and routing.
6 — LLM Provider System

Providers abstract away the LLM backend. The primary implementation uses LiteLLM as a universal gateway, with optional OAuth-based providers for services like GitHub Copilot. Provider resolution follows a cascade: explicit config, model name prefix, keyword matching, then fallback.

Provider Resolution
PRIORITY 1
Explicit provider in config
PRIORITY 2
Model prefix match (e.g. anthropic/)
PRIORITY 3
Keyword match (e.g. claude → Anthropic)
FALLBACK
First configured gateway provider
Supported Providers
  • OpenRouter Multi-provider gateway
  • Anthropic Claude models
  • OpenAI GPT + Codex models
  • DeepSeek DeepSeek R1, V3
  • Groq Fast inference + Whisper
  • Google Gemini models
  • Moonshot Kimi models
  • vLLM Self-hosted inference
  • GitHub Copilot Via OAuth provider
7 — Skills & Tools

Tools are code-level capabilities the agent can invoke (filesystem, shell, web search, etc.). Skills are markdown-based instruction extensions that shape agent behavior. Both are registered at startup and made available to the agent loop.

Built-in Tools
Each tool extends a Tool base class with name, description, parameters (JSON Schema), and execute(). The ToolRegistry collects definitions and dispatches calls by name.
  • filesystem Read, write, list files in workspace
  • shell Execute commands in the workspace
  • web_search Brave Search API
  • message Send messages to channels
  • spawn Launch subagent tasks
  • cron Schedule recurring tasks
  • mcp Invoke MCP server tools
Skills System
Skills are SKILL.md markdown files with frontmatter metadata. They can be "always-on" (injected into every prompt) or "on-demand" (activated by context). The skills engine at agent/skills.py handles discovery and loading.
  • github PR reviews, issue management
  • weather Weather lookups
  • memory Long-term memory management
  • tmux Terminal multiplexer control
  • cron Scheduled task definitions
  • clawhub Custom integrations
8 — Configuration & Entry Points

Configuration lives at ~/.nanobot/config.json, validated by Pydantic models. The CLI (Typer) provides multiple entry points for different modes of operation.

Config Structure
  • agents.defaults Model, provider, workspace, temperature
  • providers.* API keys, base URLs, headers per provider
  • channels.* Tokens, allowlists, per-channel config
  • gateway Host, port, heartbeat interval
  • tools Web search, exec, MCP servers, restrictions
Entry Points
  • nanobot agent Interactive CLI agent mode
  • nanobot agent -m "..." Single-shot message
  • nanobot gateway Multi-channel gateway server
  • python -m nanobot Module entry
  • docker compose up Containerized deployment
Gateway startup sequence. Load config → create MessageBus + ChannelManager + AgentLoop → wire CronService and HeartbeatService callbacks → run asyncio.gather(agent.run(), channels.start_all()) with background services. Defined in cli/commands.py.
