🔌 Nanobot Provider System

Comprehensive guide to how LLM providers work in Nanobot, including architecture, pathways, and practical examples.

Overview

Providers are the abstraction layer that lets Nanobot work with any LLM API — OpenAI, Anthropic, DeepSeek, custom endpoints, and more. The system automatically:

  • Detects which provider you're using
  • Prefixes model names for LLM routing (e.g., claude-opus → deepseek/claude-opus)
  • Sets up environment variables
  • Routes requests through the right code path
  • Parses responses into a uniform format

Three Provider Pathways

Providers use one of three code paths. Each is built for a specific type of LLM API.

Pathway 1: LiteLLM (Default)

RECOMMENDED

For: Any provider LiteLLM supports (OpenRouter, Anthropic, OpenAI, Gemini, DeepSeek, etc.)

How: Just add a ProviderSpec to the registry — no class needed. LiteLLM handles model prefixing, env vars, and the OpenAI SDK.

Effort: ~5 minutes (2 small edits)

Examples: Anthropic, OpenAI, Gemini, DeepSeek, DashScope

Pathway 2: CustomProvider

DIRECT

For: Self-hosted or custom OpenAI-compatible APIs (vLLM, LocalAI, Ollama)

How: Uses OpenAI Python SDK directly, bypasses LiteLLM. Set is_direct=True in ProviderSpec.

Effort: ~30 minutes (~56 lines of Python)

Code: Simple request/response parsing

Pathway 3: Custom Class

ADVANCED

For: Non-OpenAI-compatible APIs or custom auth (e.g., OpenAI Codex with OAuth)

How: Extend LLMProvider, implement custom streaming/parsing/auth

Effort: ~2 hours (~300+ lines of Python)

Code: OAuth flow, SSE parsing, tool call formatting

When to Choose Each

Scenario Choose Why
Provider is in LiteLLM's supported list Pathway 1 It's the fastest and requires no custom code
Self-hosted, OpenAI-compatible endpoint Pathway 2 CustomProvider is built for this — 56 lines, done
Non-standard auth (OAuth) Pathway 3 Need full control over auth flow
Non-OpenAI response format Pathway 3 Need custom parsing (tool calls, streaming format, etc.)

Provider Registry (PROVIDERS Tuple)

All provider configuration lives in nanobot/providers/registry.py. Adding a new provider is a 2-step process:

1

Add ProviderSpec to PROVIDERS in registry.py

Define metadata: name, keywords, API key env var, model prefix, detection rules, etc.

2

Add config field to ProvidersConfig in config/schema.py

Create the configuration class field so users can set API keys in their config file.

That's it. Everything else — env var setup, prefixing, status display — derives automatically from the spec.

ProviderSpec Fields

Click to expand the full ProviderSpec template 
ProviderSpec(
    # Identity
    name="myprovider",                     # config field name (lowercase)
    keywords=("myprovider", "mymodel"),    # case-insensitive model name keywords
    env_key="MYPROVIDER_API_KEY",          # env var for LiteLLM

    # Display
    display_name="My Provider",            # shown in `nanobot status`

    # Model Prefixing
    litellm_prefix="myprovider",           # "mymodel" → "myprovider/mymodel"
    skip_prefixes=("myprovider/",),        # don't re-prefix if already has this prefix

    # Extra env vars (e.g., for special endpoints)
    env_extras=(
        ("MY_EXTRA_VAR", "{api_key}"),     # {api_key} and {api_base} are placeholders
    ),

    # Gateway Detection (for gateways & local deployments)
    is_gateway=False,                      # can route any model? (OpenRouter, AiHubMix)
    is_local=False,                        # local deployment? (vLLM, Ollama)
    detect_by_key_prefix="sk-or-",         # match API key prefix for auto-detection
    detect_by_base_keyword="openrouter",   # match URL substring for auto-detection
    default_api_base="https://api.provider.com/v1",

    # Gateway-specific
    strip_model_prefix=False,              # strip "provider/" before re-prefixing?

    # Per-model parameter overrides
    model_overrides=(
        ("model-name", {"temperature": 1.0}),
    ),

    # Auth Type
    is_oauth=False,                        # OAuth flow instead of API key?
    is_direct=False,                       # use CustomProvider (bypass LiteLLM)?

    # Feature Support
    supports_prompt_caching=False,         # Anthropic cache_control support?
)

Model Name Resolution

LiteLLM requires provider-prefixed model names. Nanobot auto-applies prefixes from the registry — no manual prefixing needed in calling code.

Standard Provider Mode

User sets model: "claude-opus"

System detects: anthropic provider (keyword match)

Registry lookup: litellm_prefix="" (Anthropic is natively recognized)

LiteLLM gets: "claude-opus" (no prefix needed)

User sets model: "glm-4"

System detects: zhipu provider (keyword match)

Registry lookup: litellm_prefix="zai"

LiteLLM gets: "zai/glm-4" (prefix applied)

Gateway Mode

Gateways route any model, so resolution works differently:

Scenario: OpenRouter (gateway) with user model "claude-opus"

Registry: is_gateway=True, litellm_prefix="openrouter", strip_model_prefix=False

LiteLLM gets: "openrouter/claude-opus"

Scenario: AiHubMix (gateway) with user model "anthropic/claude-opus"

Registry: is_gateway=True, litellm_prefix="openai", strip_model_prefix=True

Resolution: "anthropic/claude-opus" → "claude-opus" (strip prefix) → "openai/claude-opus" (re-prefix for AiHubMix)

Why? AiHubMix is OpenAI-compatible and doesn't understand "anthropic/" prefix, so we strip it and use OpenAI prefix instead

Common Mistake: Double-Prefix

Symptom: API error mentioning "deepseek/deepseek/model" (double prefix)

Cause: User passes "deepseek/model", system adds another "deepseek/" prefix

Fix: Add the prefix to skip_prefixes:

skip_prefixes=("deepseek/",)  # Don't re-prefix if model already has this

Provider Detection & Selection

Nanobot detects which provider to use by checking, in priority order:

  1. Model name keywords — "claude" → Anthropic, "gpt" → OpenAI, etc.
  2. API key prefix — "sk-or-" → OpenRouter (for gateways)
  3. API base URL keyword — "aihubmix" in URL → AiHubMix (for gateways)
  4. Config key — provider_name from config file

Detection Code Flow

Standard providers are detected by model name via find_by_model():

def find_by_model(model: str) -> ProviderSpec | None:
    """Match by keyword (case-insensitive)."""
    model_lower = model.lower()
    
    # Check for explicit prefix match first (e.g., "deepseek/model")
    if "/" in model_lower:
        prefix = model_lower.split("/", 1)[0]
        for spec in PROVIDERS:
            if prefix == spec.name:
                return spec
    
    # Then check keywords
    for spec in PROVIDERS:
        if any(kw in model_lower for kw in spec.keywords):
            return spec
    return None

Gateways & local are detected by find_gateway():

def find_gateway(
    provider_name: str | None = None,
    api_key: str | None = None,
    api_base: str | None = None,
) -> ProviderSpec | None:
    """Detect gateway by: config key → api_key prefix → api_base keyword."""
    
    # 1. Config key wins
    if provider_name:
        spec = find_by_name(provider_name)
        if spec and (spec.is_gateway or spec.is_local):
            return spec
    
    # 2. API key prefix (e.g., "sk-or-" → OpenRouter)
    for spec in PROVIDERS:
        if spec.detect_by_key_prefix and api_key:
            if api_key.startswith(spec.detect_by_key_prefix):
                return spec
    
    # 3. API base URL keyword (e.g., "aihubmix" in URL)
    for spec in PROVIDERS:
        if spec.detect_by_base_keyword and api_base:
            if spec.detect_by_base_keyword in api_base:
                return spec
    
    return None

LLMResponse Contract

All providers normalize their API response into LLMResponse — a universal envelope.

Data Structure

@dataclass
class LLMResponse:
    content: str | None                    # Text response (None if tool-calls only)
    tool_calls: list[ToolCallRequest]      # Parsed tool call requests
    finish_reason: str                     # "stop", "tool_calls", "error", etc.
    usage: dict[str, int]                  # prompt_tokens, completion_tokens, total_tokens
    reasoning_content: str | None          # Chain-of-thought (DeepSeek-R1, Kimi K2.5)
    thinking_blocks: list[dict] | None     # Structured thinking (Anthropic extended thinking)
    
    @property
    def has_tool_calls(self) -> bool:
        return len(self.tool_calls) > 0


@dataclass
class ToolCallRequest:
    id: str              # Short unique ID (9-char alphanumeric)
    name: str            # Tool function name
    arguments: dict      # Parsed arguments (NOT raw JSON string)

Normalization Rules

  • content: use None (not "") when assistant is tool-calling with no text
  • reasoning_content: use or None pattern to convert empty strings to None
  • thinking_blocks: use or None pattern (same reason)
  • usage keys must always be standard: prompt_tokens, completion_tokens, total_tokens
  • Both thinking fields are persisted to session history for future context

Example 2: Using CustomProvider for vLLM

You have a local vLLM server at http://localhost:8000/v1 with Llama-2 model.

Configuration

[agents]
model = "Llama-2-7B-chat"

[providers.custom]
api_key = "no-key"
api_base = "http://localhost:8000/v1"

How It Works

  • Config key custom is in registry with is_direct=True
  • System uses CustomProvider instead of LiteLLM
  • CustomProvider uses OpenAI SDK directly (vLLM is OpenAI-compatible)
  • Request goes: CustomProvider.chat() → AsyncOpenAI(...).chat.completions.create() → vLLM
  • Response is parsed into LLMResponse

Why CustomProvider?

vLLM is OpenAI-compatible, so it doesn't need LiteLLM's multi-provider abstraction. CustomProvider is lighter and faster.

Example 3: Detecting OpenRouter with API Key

User sets OPENROUTER_API_KEY=sk-or-xxx. How does Nanobot detect it?

Detection Flow

  1. LiteLLMProvider.__init__() calls find_gateway(provider_name, api_key, api_base)
  2. No provider_name set, so check API key prefix
  3. API key starts with "sk-or-"
  4. Registry lookup finds OpenRouter spec with detect_by_key_prefix="sk-or-"
  5. _gateway = OpenRouter spec
  6. System knows it's a gateway, applies openrouter/ prefix to models

Usage

# User can now use ANY model through OpenRouter:
provider = LiteLLMProvider(
    api_key="sk-or-xxx",
    default_model="claude-opus"
)

# Behind the scenes:
# 1. Detects OpenRouter gateway automatically
# 2. Resolves "claude-opus" → "openrouter/claude-opus"
# 3. LiteLLM routes it through OpenRouter API
# 4. OpenRouter calls the actual Claude API
# 5. Response normalized to LLMResponse

Environment Variables & Setup

The registry automatically manages all env vars from ProviderSpec metadata.

Standard Pattern

ProviderSpec: env_key="ANTHROPIC_API_KEY"

User sets: ANTHROPIC_API_KEY=sk-ant-xxx

LiteLLMProvider: Reads from env and passes to LiteLLM

LiteLLM: Uses it to authenticate with Anthropic API

Extra Env Vars (env_extras)

Some providers need additional env vars beyond the API key. Example: Moonshot's MOONSHOT_API_BASE:

ProviderSpec(
    name="moonshot",
    env_key="MOONSHOT_API_KEY",
    env_extras=(
        ("MOONSHOT_API_BASE", "{api_base}"),  # Placeholder: user's api_base or spec default
    ),
)

Placeholder rules:

  • {api_key} → User's API key
  • {api_base} → User's config api_base, or spec.default_api_base

Advanced: Adding a Custom Provider Class

Only needed when LiteLLM doesn't support the provider AND the API is not OpenAI-compatible. This is rare.

When to Do This

  • Provider's API format is not OpenAI-compatible
  • Provider uses OAuth instead of API keys
  • Provider has custom streaming format for tool calls
  • Provider has unique tool call ID format that breaks compatibility

Real Example: OpenAICodexProvider

OpenAI Codex uses OAuth (not API keys) and custom SSE streaming format. It extends LLMProvider with ~314 lines:

class OpenAICodexProvider(LLMProvider):
    """Custom OAuth-based provider for OpenAI Codex."""
    
    def __init__(self, api_base: str = "https://chatgpt.com/backend-api"):
        super().__init__()
        self.api_base = api_base
        self._session = None  # For OAuth flow & cookies
    
    async def chat(self, messages: list[dict[str, Any]], ...):
        """Implement OAuth flow, custom request format, SSE parsing."""
        # 1. Get OAuth token (user login flow)
        # 2. Build custom message format (not standard OpenAI)
        # 3. Stream response as Server-Sent Events
        # 4. Parse tool calls from custom format
        # 5. Return LLMResponse
        pass
    
    def get_default_model(self) -> str:
        return "gpt-4-turbo"

How to Implement

  1. Extend LLMProvider — inherit base class
  2. Implement async def chat() — handle request/response
  3. Implement get_default_model() — default model name
  4. Normalize response to LLMResponse — use ToolCallRequest for tool calls
  5. Add ProviderSpec with is_oauth=True or custom detection
  6. Register in _make_provider() in CLI commands

Expected time: 1–2 hours for a complete implementation with testing.

Architecture Decisions

Why Registry Pattern?

  • Single source of truth — all provider metadata in one place
  • DRY principle — env vars, prefixing, detection all derive from spec
  • No if-elif chains — lookup instead of conditionals
  • Auto-discovery — nanobot status shows all providers without extra code
  • Extensible — add a spec in 5 minutes, no class needed

Why Three Pathways?

  • LiteLLM — covers 95% of use cases, minimal boilerplate
  • CustomProvider — 56 lines for OpenAI-compatible endpoints, avoids LiteLLM overhead
  • Custom Class — full control for edge cases (OAuth, custom streaming, etc.)

Why LLMResponse Contract?

  • Decoupling — agent loop doesn't care which provider was used
  • Uniform handling — tool calls are parsed identically regardless of provider
  • Persistence — reasoning content & thinking blocks saved to session history
  • Debugging — always know what the LLM returned in standard format

Debugging Provider Issues

Common Issues & Solutions

Error: "Model not found" or "API error"

Likely cause: Model name not matched to provider.

Debug:

  1. Run nanobot status to see detected provider and resolved model
  2. Check if provider is configured (API key set)
  3. Verify model name keywords match registry
  4. If using a gateway, ensure prefix is being applied correctly
Error: Double-prefix (e.g., "deepseek/deepseek/model")

Likely cause: Model already has prefix, system added another.

Fix: Add prefix to skip_prefixes in ProviderSpec

skip_prefixes=("deepseek/",)  # Skip re-prefixing
Error: Environment variable not found

Debug:

  1. Check nanobot status for API key configuration
  2. Verify env var name matches registry (e.g., ANTHROPIC_API_KEY)
  3. For gateways, check key prefix matches detect_by_key_prefix
  4. For extra env vars, verify placeholders are resolved (check logs)
Wrong provider detected with same API key

Likely cause: Multiple providers share same env var, detection order matters.

Fix: Use detect_by_key_prefix to distinguish (e.g., OpenRouter's "sk-or-")

Or: Explicitly set provider_name in config to force detection

Summary

The Provider System at a Glance

  • Registry-driven: All provider metadata in PROVIDERS tuple — single source of truth
  • Three pathways: LiteLLM (default), CustomProvider (OpenAI-compatible), custom class (advanced)
  • Auto-detection: By model name, API key prefix, or config key
  • Auto-prefixing: Model names automatically prefixed for LiteLLM routing
  • Unified response: All providers normalize to LLMResponse
  • Extensible: Add a new provider in 5 minutes (just ProviderSpec + config field)

Key Files

  • nanobot/providers/registry.py — ProviderSpec definitions & lookup logic
  • nanobot/providers/base.py — LLMProvider abstract base, LLMResponse contract
  • nanobot/providers/litellm_provider.py — Default pathway (LiteLLM-based)
  • nanobot/providers/custom_provider.py — Direct pathway (OpenAI SDK)
  • nanobot/config/schema.py — ProvidersConfig for user configuration

Quick Start: Adding a New Provider

  1. Check if LiteLLM supports it: if yes, use Pathway 1
  2. If OpenAI-compatible: use Pathway 2
  3. Otherwise: implement custom class extending LLMProvider
  4. Add ProviderSpec to PROVIDERS tuple in registry.py
  5. Add config field to ProvidersConfig in config/schema.py
  6. Run nanobot status to verify detection
