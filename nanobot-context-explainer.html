Nanobot Context System

How identity, memory, and skills are assembled into agent prompts

1

System Overview

The nanobot context system is the brain of the agent. It builds comprehensive prompts by layering identity, environment metadata, persistent memory, and available skills. The ContextBuilder class orchestrates this assembly process.

Core Purpose

Construct complete message lists (system prompt + conversation history) for LLM calls, ensuring the agent has full situational awareness.

Key Inputs

Identity, bootstrap files (AGENTS.md, SOUL.md, USER.md, TOOLS.md, IDENTITY.md), memory store, available skills, and conversation history.

Key Output

A structured message array ready for LLM consumption: [system, ...history, runtime_context, user_message].

Key Design Principle

Context is immutable during a single turn but rebuilt fresh for each new turn. This ensures the agent always sees current memory, time, and skills while preventing context pollution from previous interactions.

2

Context Components

The context is composed of five distinct layers, each providing different types of guidance and information to the agent.

Identity Layer

What: Agent name, runtime environment, workspace path, guidelines.

Example: "You are nanobot, a helpful AI assistant running on macOS Arm64 with Python 3.11."

Purpose: Grounds the agent in reality with hardware, OS, and core personality.

Bootstrap Files

What: Markdown documents loaded from workspace root (AGENTS.md, SOUL.md, USER.md, TOOLS.md, IDENTITY.md).

Example: SOUL.md might contain "You prioritize clarity over cleverness" and "Always ask for confirmation before destructive operations."

Purpose: Inject project-specific values, constraints, and patterns.

Memory Layer

What: Long-term facts extracted from MEMORY.md (last N entries within a token window).

Example: "The user prefers async/await over promises. The codebase uses Jest for testing."

Purpose: Preserve cross-session context without storing entire conversation history.

Skills Layer

What: Always-enabled skills (full content) + summary of on-demand skills (names + descriptions).

Example: Always-enabled: "visual-explainer skill for generating HTML diagrams." On-demand: "ast-grep for advanced code search."

Purpose: Advertise capabilities and provide implementation guidance for available tools.

Runtime Context

What: Current time, timezone, channel/chat metadata (injected before user message).

Example: "[Runtime Context] Current Time: 2026-03-01 14:35 (Sunday)"

Purpose: Provide temporal grounding and message routing information without polluting the main system prompt.

Assembly Code Walkthrough

build_system_prompt() sequence
1. Get identity (runtime, workspace paths, guidelines) 2. Load bootstrap files (AGENTS.md, SOUL.md, USER.md, TOOLS.md, IDENTITY.md) 3. Append memory context (MEMORY.md, last N entries) 4. Append always-enabled skills (full content) 5. Append skills summary (names + descriptions of on-demand tools) 6. Join all parts with "---" separator Result: A single comprehensive system prompt string

Why Layering Matters

Identity first: Establishes base personality and reality grounding. Bootstrap second: Project-specific overrides. Memory third: Contextual facts from prior sessions. Skills last: Capability advertisement doesn't overshadow values or facts.

System Prompt Structure

A complete system prompt is a multi-part document. Each section plays a specific role in shaping agent behavior.

Identity Section

_get_identity()

Platform (macOS Arm64, Python 3.11), workspace path, long-term memory location, history log location, and nanobot guidelines.

# nanobot 🐈
You are nanobot, a helpful AI assistant.

## Runtime
macOS arm64, Python 3.11

Bootstrap Section

_load_bootstrap_files()

Reads all present bootstrap files from workspace root and includes their full content.

AGENTS.md SOUL.md USER.md TOOLS.md IDENTITY.md

Memory Section

memory.get_memory_context()

Last N lines of MEMORY.md within the token window. Provides persistent facts across sessions.

# Memory
[2026-02-28 10:30] User prefers async/await.
[2026-02-27 14:22] Project uses Jest.

Skills Sections

skills.get_always_skills() + summary

Active skills (full SKILL.md content) + on-demand summary (names, descriptions).

Message Order Matters

The system prompt is always first, then conversation history, then runtime context (injected fresh each turn), then the current user message.

This ensures the agent sees the most recent time, channel, and user intent without losing context from previous exchanges.

Tool Result Handling

add_tool_result() appends tool execution results back to the message array.

The agent uses these results to continue reasoning or make follow-up tool calls in the next iteration.

Media Handling

The _build_user_content() method optionally encodes images as base64 and includes them in the user message for vision-capable models:

Fresh Context Every Turn

The system prompt is rebuilt for each LLM call, not cached. This means memory updates, new skills, and bootstrap file changes are reflected immediately without restarting the agent. It's a deliberate design choice for responsiveness.

AgentLoop Usage Pattern

How AgentLoop uses ContextBuilder
# In AgentLoop.__init__ self.context = ContextBuilder(workspace) # In AgentLoop.process_message() for iteration in range(max_iterations): # Build fresh system prompt (includes current memory) system_prompt = self.context.build_system_prompt() # Build complete message list messages = self.context.build_messages( history=session.messages, current_message=user_input, skill_names=None, # Optional skill filtering media=attached_images, channel=channel_name, chat_id=chat_id ) # Call LLM response = self.provider.complete( model=self.model, messages=messages, temperature=self.temperature, max_tokens=self.max_tokens ) # If tool calls, add results and loop if response.tool_calls: for tool_call in response.tool_calls: result = execute_tool(tool_call) messages = self.context.add_tool_result( messages, tool_call_id=tool_call.id, tool_name=tool_call.name, result=result )

Practical Code Examples

Real-world examples of how to use the ContextBuilder for different scenarios.

Example 1: Basic Context Building

Python: Build context and make LLM call
from pathlib import Path from nanobot.agent.context import ContextBuilder workspace = Path("/Users/jim.salinas/projects/nanobot") context = ContextBuilder(workspace) # Build system prompt (includes identity, bootstrap, memory, skills) system_prompt = context.build_system_prompt() print(f"System prompt length: {len(system_prompt)} chars") # Build complete message list messages = context.build_messages( history=[], # Start with no history current_message="What is in the nanobot context system?" ) # Send to LLM # response = provider.complete(model="claude-3-5-sonnet", messages=messages)

Example 2: With Conversation History

Python: Building messages with prior exchanges
messages = context.build_messages( history=[ { "role": "user", "content": "Tell me about context building" }, { "role": "assistant", "content": "The ContextBuilder assembles identity, bootstrap files, memory, and skills..." } ], current_message="Can you explain the message structure?", channel="slack", chat_id="C123456" ) # Result includes: # 1. System prompt (built fresh) # 2. History (prior exchanges) # 3. Runtime context (timestamp, channel, chat_id) # 4. Current message (what the user just asked)

Example 3: With Images

Python: Including vision content
messages = context.build_messages( history=[], current_message="What do you see in this screenshot?", media=[ "/path/to/image.png", "/path/to/screenshot.jpg" ] ) # The _build_user_content() method: # 1. Reads each file # 2. Checks if it's a valid image type # 3. Base64-encodes the content # 4. Builds an image_url block with data URI # 5. Appends the text message # # Result: Vision-capable models can see and analyze the images

Example 4: Tool Result Handling

Python: Adding tool results in a loop
# Initial LLM call response = provider.complete(model="claude", messages=messages) # Agent decides to call a tool if response.tool_calls: for tool_call in response.tool_calls: # Execute the tool tool_result = shell_tool.execute(tool_call.input) # Add assistant message with tool call messages = context.add_assistant_message( messages, content=response.content, tool_calls=response.tool_calls ) # Add tool result messages = context.add_tool_result( messages, tool_call_id=tool_call.id, tool_name=tool_call.name, result=tool_result ) # Now messages contains the tool result and agent can reason about it # Continue loop for next LLM call with updated messages

Example 5: Building with Specific Skills

Python: Filtering skills by name
# Build context with only specific skills enabled skill_names = ["visual-explainer", "ast-grep"] system_prompt = context.build_system_prompt(skill_names=skill_names) # Note: In current implementation, skill_names parameter exists but # actual filtering would happen in build_messages() messages = context.build_messages( history=[], current_message="Generate a diagram for this architecture", skill_names=skill_names # Optional: filter skills in future versions ) # The system prompt now highlights specific skills relevant to the task

Pro Tips

  • ✓ Fresh rebuild: Always rebuild the system prompt for each turn to get updated memory
  • ✓ Message immutability: The message list grows with each tool call; don't reuse stale lists
  • ✓ Runtime context: Injected automatically; you only need to pass current_message
  • ✓ Memory updates: Write to MEMORY.md within tool execution; next turn sees it
  • ✓ Bootstrap changes: Edit AGENTS.md, SOUL.md, etc.; next turn rebuilds with new content
