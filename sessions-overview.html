Session Management in Nanobot

Understanding conversation persistence, memory consolidation, and history management

What is a Session?

A session is a persistent conversation context that tracks all messages exchanged between a user and the agent within a specific channel and chat. Sessions enable multi-turn conversations where the agent remembers previous interactions and can reference past context.

Session Identification

Sessions are uniquely identified by a session_key in the format:

"channel:chat_id"

Example: slack:user-1234 or discord:channel-5678

Data Persistence

Sessions are stored as JSONL files in the sessions/ directory within the workspace:

workspace/sessions/{safe_key}.jsonl

Each line is a JSON object (message or metadata).

Core Components

Session Data Structure

Each session is represented by a dataclass with:

Field Type Purpose
key str Unique identifier (channel:chat_id)
messages list[dict] Append-only conversation history
created_at datetime Session creation timestamp
updated_at datetime Last update timestamp
metadata dict Custom session metadata
last_consolidated int Number of messages consolidated to memory files

SessionManager

Manages the lifecycle of all sessions:

  • Cache: In-memory dictionary for fast lookup
  • Persistence: Load/save to JSONL files
  • Legacy support: Migrate sessions from ~/.nanobot/
  • Discovery: List all available sessions

Message History

get_history(max_messages=N) returns unconsolidated messages:

  • Latest N messages from session
  • Aligned to user turn (no orphaned tools)
  • Used as context for LLM
  • Never modifies original messages

Message Flow in Detail

Step-by-step: Processing a Message

1. Receive & Extract

InboundMessage arrives from channel bus. Extract channel and chat_id to construct session_key.

2. Get/Create Session

SessionManager.get_or_create(key) checks cache, then disk. Creates new Session if needed.

3. Check Commands

/new clears session after consolidating memory. /help returns command list.

4. Retrieve History

session.get_history(max_messages=100) fetches unconsolidated messages for context.

Important: The get_history() method returns unconsolidated messages only. Messages that have been archived to memory files (via consolidation) are not included, preventing duplication of summarized context.

5. Build Context

ContextBuilder creates full message list with history + current message + system context.

6. Run Agent Loop

LLM processes context, makes tool calls, generates response.

7. Save Messages

_save_turn() appends new messages to session, truncating large tool results.

8. Persist Session

SessionManager.save() writes entire session to JSONL file on disk.

Memory Consolidation

The Problem: Context Window Limits

As conversations grow longer, the full message history becomes too large to fit in the LLM's context window. Memory consolidation solves this by archiving old messages to summary files while keeping the session lightweight.

Consolidation Triggers

  • Automatic: when unconsolidated ≥ memory_window
  • /new command: consolidate all before clearing
  • Runs async to avoid blocking message processing

Key Principle: Append-Only

Messages are never modified or deleted. Consolidation only updates the last_consolidated pointer, telling get_history() where to start reading.

LLM Cache Efficiency: Because messages are append-only, the LLM's token cache (supported by OpenAI, Anthropic, etc.) remains valid. Resuming a conversation reuses the cached prompt tokens, saving cost and latency.

Tool Context & Routing

The agent loop calls _set_tool_context() at the start of message processing to inject routing information into tools that need it.

Context-Aware Tools

Tool Context Used Why
MessageTool channel, chat_id, message_id Routes responses back to correct conversation thread
SpawnTool channel, chat_id Subagents inherit parent session/channel context
CronTool channel, chat_id Scheduled tasks reference their triggering context

This ensures that when tools execute, they know which conversation they're operating in and can send results back to the right place.

Storage: JSONL Format

Sessions are stored as JSONL (JSON Lines) files—one JSON object per line. This format is efficient, human-readable, and plays nicely with streaming and line-based tools.

File Structure

{"_type":"metadata","key":"slack:user-123","created_at":"2024-01-15T10:30:00","updated_at":"2024-01-15T11:45:00","last_consolidated":12,"metadata":{}}
{"role":"user","content":"What is nanobot?","timestamp":"2024-01-15T10:30:01"}
{"role":"assistant","content":"Nanobot is an agentic system...","timestamp":"2024-01-15T10:30:05"}
{"role":"user","content":"Can it handle long conversations?","timestamp":"2024-01-15T10:35:02"}
{"role":"assistant","content":"Yes, through memory consolidation...","timestamp":"2024-01-15T10:35:06"}
...

Metadata Line

First line contains session metadata with _type: "metadata" marker for easy parsing.

Message Lines

Subsequent lines are messages with role, content, timestamp, and optional tool_calls or tool_call_id.

Safe Filenames: The session key is converted to a safe filename (colons replaced with underscores, special chars removed) to work reliably across filesystems.

Concurrency & Thread Safety

Processing Lock

_processing_lock serializes all incoming messages globally, ensuring one message is processed at a time (no race conditions).

Consolidation Locks

Per-session locks prevent concurrent consolidation on the same session, but allow parallel consolidation across different sessions.

Active Tasks Tracking

_active_tasks maintains a mapping of session keys to their running asyncio tasks. This allows graceful cancellation via /stop command.

Graceful Shutdown: When /stop is received, the agent cancels all active tasks for that session and waits for subagents to finish, ensuring clean termination.

  • get_or_create(key) — Get or create a session

  • invalidate(key) — Remove from in-memory cache
  • list_sessions() — Discover all sessions

Summary

🔑 Identification

Sessions are keyed by channel:chat_id and cached in memory for fast access.

💾 Persistence

Stored as JSONL files in workspace/sessions/, with append-only message lists.

🧠 Memory

Consolidation archives old messages to summary files when conversation grows too long.

Core Design Principle: Sessions are immutable conversation records. New messages are always appended; old messages are never deleted or modified. This ensures LLM cache efficiency, easy auditing, and reliable history reconstruction.
